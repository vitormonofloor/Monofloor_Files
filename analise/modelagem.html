Aba HOJE — espelho client-side de /api/projects

VEREDITO PROVADO BYTE-A-BYTE: abrir uma obra em /app/hoje retorna o MESMO payload de /app/projetos/{id}. Auditor testou obra ca4b3047... em /api/projects/{id} e /api/projects/{id}?context=hoje — 8.433 bytes, 113 chaves, idênticos. Query ?context=hoje é ignorada. Backend único.

Como a aba HOJE é montada

AspectoDetalhe
RenderizaçãoSSR puro Next.js App Router (1,58 MB de HTML)
Sem __NEXT_DATA__RSC streaming — não há JSON estruturado exposto na página
Fonte de dados/api/projects + /api/escalacao-diaria + /api/whatsapp/conversations
Filtro temporalderivação client-side de campos já existentes (dataInicio, data_e_hora_da_visita, data_de_entrada, dias_e_hor_rio_para_entrega...)
Real-timepolling setInterval(10s) sobre /api/whatsapp/* — sem WebSocket/SSE
Custo~104 KB a cada 10s = ~37 MB/h por usuário logado, sem ETag
Rota da obra individual/app/projeto/{uuid} (singular, não plural)

Anomalias específicas da aba HOJE

Vazamento LGPD: /app/hoje retorna 200 em GET anônimo — expõe nomes de clientes, aplicadores e UUIDs publicamente sem auth
Bug 500: /api/projects/today retorna {"error":"Failed to fetch project"} — handler trata "today" como UUID sem validação
Polling pesado: 104 KB a cada 10s sem ETag/If-Modified-Since — ~37 MB/h por usuário só pra atualizar conversas
Duas fontes para a mesma data: dataExecucaoPrevista (top-level) convive com dataInicio/dataTerminoPrevista (obraMap) — risco de divergência silenciosa
Logística depende de string livre: "dias_e_hor_rio_para_entrega" não é timestamp — typo humano remove o evento da aba HOJE
TZ frágil: "chegadasHoje" sem fuso explícito — chegadas após 21h BRT podem virar "amanhã" se query roda em UTC
Typos estruturais duplicados: cofirmar_data_de_entrada (sem 'n') · confirmar_data_de_entradaa (com 'aa') · confirmar_endere_oo — migração Pipefy não limpa
Rótulos UI redundantes: "Em obra - execução" (hífen) vs "Em obra – execução" (travessão) — concatenação de fontes distintas
Não é Telegram, é WhatsApp: stack 100% WhatsApp; nenhuma referência a Telegram nos chunks JS
responsavel = null em 100% das conversas WhatsApp amostradas — roteamento de atendimento não preenchido
Sem feed/galeria de fotos global: imagens só vivem dentro de /api/whatsapp/messages com messageType=image · mediaUrl

Lógica de cálculo

Fórmula observada

dataFim_obra = dataInicio + diasTotal(úteis), considerando trabalhaSabado/Domingo
diasTotal ≠ diasInstalacao + diasCura (campos legados, <17% preenchidos). Verdade vive em etapas[] — cada etapa duração=1d, sequencial sem folga (dataInicio[N] = dataFim[N-1] + 1).

Anomalias específicas do Planejamento

Cronograma fossilizado: 33/33 obras ativas com dataTermino < hoje — nunca re-planejado após atraso
97% das etapas sem prestadoresIds: alocação não persistida; operação feita FORA do sistema
100% dos 1.037 projects com dataExecucaoConfirmada=false — flag nunca atualizado via API
49% dos projects com diasTotal=0 — estimativa não calculada
Sem campo de restrição climática (contradiz status aguardando_clima)
Sem feriado/calendário parametrizado
Sem dependência entre obras — cada obra é ilha; sem fila/grupo/predecessora
